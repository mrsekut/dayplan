# v2 プロトタイピングで見えた未解決課題

2026-04-09 のプロトタイピングセッションで発見した、まだ答えが出ていない設計課題。

## 1. complete後に次のタスクが始まらない問題

現状: completeすると同じ作業枠内の次のpendingがactivateされる。
枠内にpendingがなければ何も起きない。次の作業枠のタスクは自動で始まらない。

単純に「plan全体から次のpendingをactivate」にすると、
グルーピング（雑務まとめ消化等）が壊れる。

## 2. 「作業枠」はユーザーに見えるべき概念か？

作業枠 = 固定ブロック間の空き時間（実装詳細）。
しかしタスクのグルーピングの器としても機能している。

- ユーザーが意識すべき？ → 「この30分は雑務消化」というグルーピング
- 実装詳細として隠すべき？ → 「タスクキューがあって順番に来る」だけ

両方の性質がある。分離すべきかもしれない:
- 「時間スロット」: 固定ブロック間の時間（自動生成、実装詳細）
- 「タスクグループ」: ユーザーが意図的に作るまとまり（雑務バッチ等）

## 3. 大きいissueの分割

1.5hのissueは30分単位で分割すべき。
これはdayplanの責務ではなくAI側のskill。
