# Dayplan v2 PRD

## 問題の定義

日々のタスク管理において、朝にスケジュールを組んでも、差し込み（Slack対応等）が発生するたびにリスケの手間が大きく、スケジュールが現実と乖離して形骸化する。結果として「次に何をやるか」の意思決定コストが削減できず、個々のタスクへの集中が妨げられている。

具体的な痛み:

- リスケの摩擦が大きい（JSON再生成やAIへの指示が重い）
- タスク完了の明示的なマークがなく、時間オーバーを無視してしまう
- 5分で終わる細かいタスクの居場所がない（スケジュールに載せるには細かすぎ、載せないと忘れる）
- PiP表示は見えるが行動を変えるほどの強制力がない

## 解決策

1日を「動かせない予定（MTG等の固定ブロック）」と「作業枠（固定ブロック間の空き時間）」に分け、作業枠の中はタスクのキュー（順番待ち行列）として管理する。

- 固定ブロックは絶対に動かない。MTG、面接準備など時刻指定が必要なものはここに入る
- 作業枠内のタスクには厳密な開始・終了時刻がない。順番に消化する
- タスク完了は明示的にマークする。未マークなら次に進めない
- 細かいタスクはバッチとしてグルーピングし、1つの作業枠内でまとめて消化する
- リスケという概念をなくす。作業枠内では順番を変えるか、タスクを追加するだけ

アーキテクチャはレイヤーを疎結合にする:

- **Core**: スケジュールデータ操作。UIやCLIに依存しない
- **CLI**: AI向け + 最低限の人間操作。Coreを呼ぶだけ
- **UI**: 人間向け表示（TUI / Web / デスクトップアプリ）。後で別途開発

## ユーザーストーリー

### スケジュール作成

1. AIエージェントとして、Googleカレンダーの固定予定とタスクリストからDayPlanをJSON生成して `set` コマンドで投入したい。なぜなら朝のプランニングを自動化できるからだ。
2. ユーザーとして、固定ブロック（MTG等）を指定すると、間の空き時間が自動で作業枠になってほしい。なぜなら手動で空き時間を計算したくないからだ。
3. ユーザーとして、細かいタスク（5-10分）をバッチとしてまとめて1つの作業枠に入れたい。なぜなら個別にスケジューリングするには粒度が細かすぎるからだ。
4. ユーザーとして、「午前中にやりたい」「集中力が要る」などの性質に基づいてタスクを適切な作業枠に配置してほしい。なぜなら思考系タスクは午前にやりたいからだ。（※ 配置のロジック自体はAIに任せる。ツール側はタスクのkindを保持するだけ）

### タスク消化

5. ユーザーとして、現在のタスクと作業枠の残り時間を確認したい。なぜなら次のMTGまでの時間感覚を把握したいからだ。
6. ユーザーとして、タスクを完了マークしたら自動で次のタスクが提示されてほしい。なぜなら意思決定なしに次の作業に入りたいからだ。
7. ユーザーとして、今のタスクをスキップして次に進みたい。なぜなら気分や状況に応じて順番を変えたいことがあるからだ。
8. ユーザーとして、作業枠が終わっても残ったタスクは次の作業枠に自動繰越してほしい。なぜなら手動で移動するのは面倒だからだ。

### 差し込み対応

9. ユーザーとして、差し込みタスクをその場でキューに追加したい。なぜならSlack対応などが突発的に発生するからだ。
10. AIエージェントとして、差し込み発生時にキューの並び替えを提案したい。なぜならユーザーの代わりに優先度判断を行えるからだ。

### AI連携

11. AIエージェントとして、全コマンドで `--json` オプションにより構造化された出力を得たい。なぜなら出力をパースして次のアクションを判断する必要があるからだ。
12. AIエージェントとして、`prime` コマンドで現在のスケジュール全体のコンテキストを取得したい。なぜならセッション開始時に状況を把握する必要があるからだ。

## 実装に関する決定事項

### ランタイム・技術スタック

- Bun（Node.jsではない）
- TypeScript
- テスト: Bun の組み込みテストランナー

### データモデル

```typescript
type TaskKind = 'focus' | 'batch' | 'mtg' | 'other';

type FixedBlock = {
  id: string;
  type: 'fixed';
  start: string; // "HH:MM"
  end: string; // "HH:MM"
  title: string;
  kind: TaskKind;
};

type Task = {
  id: string;
  title: string;
  estimatedMinutes: number;
  kind: TaskKind;
  status: 'pending' | 'active' | 'completed' | 'skipped';
  beadId?: string; // 将来: Beads連携
};

type WorkSlot = {
  id: string;
  type: 'work';
  start: string; // "HH:MM"（固定ブロックから自動算出）
  end: string; // "HH:MM"（固定ブロックから自動算出）
  queue: Task[];
};

type DayPlan = {
  date: string; // "YYYY-MM-DD"
  entries: (FixedBlock | WorkSlot)[];
};
```

### モジュール構成

| モジュール    | 責務                                                                                     | テスト |
| ------------- | ---------------------------------------------------------------------------------------- | ------ |
| `schedule.ts` | データモデル定義、タスク操作（complete, skip, activate, carry-over）、時刻ユーティリティ | あり   |
| `planner.ts`  | 固定ブロック + タスクリスト → DayPlan生成。作業枠の自動算出                              | あり   |
| `storage.ts`  | ファイルベースJSON永続化（`~/.config/dayplan2/YYYY-MM-DD.json`）                         | なし   |
| `cli.ts`      | コマンドルーティング、stdin/stdout、`--json`対応                                         | なし   |

### CLIコマンド

```
dayplan set              # DayPlanをstdinのJSONから設定
dayplan show [date]      # スケジュール表示
dayplan status           # 現在のタスク + 作業枠の残り時間
dayplan complete         # 現在のタスクを完了、次を提示
dayplan skip             # 現在のタスクをスキップ
dayplan add-task         # タスクをキューに追加（JSON from stdin）
dayplan prime            # AI向けフルコンテキスト出力
dayplan help             # ヘルプ
```

全コマンドで `--json` フラグをサポートし、構造化出力を返す。

### setコマンドの入力フォーマット

```json
{
  "date": "2026-04-10",
  "fixedBlocks": [
    { "start": "11:30", "end": "12:00", "title": "1on1", "kind": "mtg" },
    { "start": "15:00", "end": "16:00", "title": "採用面接", "kind": "mtg" }
  ],
  "tasks": [
    { "title": "機能Aの設計", "estimatedMinutes": 60, "kind": "focus" },
    { "title": "Slack返信まとめ", "estimatedMinutes": 30, "kind": "batch" }
  ]
}
```

固定ブロック間の空き時間を自動で作業枠として生成し、タスクを振り分ける。作業日の範囲はデフォルト09:00-18:00（将来的に設定可能にするかもしれないがスコープ外）。

### アーキテクチャ方針

- package by feature（`core/` のようなレイヤーディレクトリは作らない）
- UIレイヤーは疎結合。Core/CLIはUIに依存しない
- 操作関数は純粋関数（DayPlanを受け取り新しいDayPlanを返す）にする。副作用はCLI層に閉じる

## テストに関する決定事項

### 良いテストとは

- 関数の外部から見た振る舞いをテストする（入力→出力）
- 内部実装の詳細（IDの値、中間状態）に依存しない
- エッジケース（空のキュー、全タスク完了済み、作業枠をまたぐケース等）をカバーする

### テスト対象モジュール

**schedule.ts:**

- completeCurrentTask: 完了→次タスクがactiveになること
- skipCurrentTask: スキップ→次タスクがactiveになること
- carryOverTasks: 過ぎた作業枠の未完了タスクが次の枠に移ること
- activateNextTask: pending→activeの遷移
- 全タスク完了時の挙動

**planner.ts:**

- 固定ブロックから作業枠が正しく生成されること
- 固定ブロックが隣接している場合（空き時間なし）
- タスクの振り分け（作業枠の容量を超えた場合の挙動）
- 固定ブロックなしの場合（1日全体が1つの作業枠）

### テストの先行事例

Proto Bの `proto-b/src/core/schedule.ts` が参考になる。また旧dayplanの `test/core/schedule.test.ts` にも類似のテストパターンがある。

## スコープ外

- **UI（TUI / Web / デスクトップアプリ）**: 今回はCore + CLIのみ。UIは別途開発
- **通知**: macOS通知、PiP表示などは後で追加
- **Beads連携**: `beadId`フィールドは用意するが、実際の連携ロジックは後で
- **Strand連携**: AI自動実行との統合は後で
- **振り返り自動化**: 1日の終わりの自動振り返りは優先度低
- **作業日の範囲設定**: 09:00-18:00固定。設定可能にするのは後で
- **画面キャプチャによる脱線検知**: 将来的な理想形だが今回はスコープ外

## 補足事項

- 旧dayplanの「AI向けインターフェース」の良さ（prime, onboard, --json）は踏襲する
- タスクのバッチグルーピング（細かいタスクをまとめる）はAI側の責務。ツール側はkind: 'batch'を保持するだけで、グルーピングロジックは持たない
- 「午前に思考系、午後に作業系」などの配置もAI側の責務。ツール側はタスクのkindと作業枠の時間帯を提供するだけ
